**Fecha:** 2026-07-09

## 1. Executive Summary

MIleGo V7 Phase 1 entregó la plataforma completa: API multi-tenant, auth, productos, categorías, usuarios, admin SPA, event bus, jobs, auditoría y storage. Antes de agregar la capa de negocio (órdenes, clientes, pagos, inventario), este sprint **no agrega features**. Corrige, asegura, documenta y prueba lo que ya existe.

### Principios del Hardening Sprint

1. **Seguridad sobre features** — Una vulnerabilidad multi-tenant anula cualquier feature nuevo.
2. **Contrato sobre implementación** — OpenAPI no es opcional; es el pacto entre frontend, admin e integraciones.
3. **Test sobre intuición** — Si no hay test automatizado, no está listo para producción.
4. **Recuperabilidad sobre velocidad** — Jobs fallidos y eventos perdidos son más costosos que procesos lentos.

### Criterio de cierre

El sprint está completo cuando:

- ✅ Una tienda no puede acceder a datos de otra (test automatizado)
- ✅ Tokens revocados no funcionan (incluyendo refresh rotation abuse)
- ✅ API respeta el contrato OpenAPI documentado
- ✅ Jobs fallidos son recuperables (dead letter queue)
- ✅ Eventos no se pierden (persistencia + replay)
- ✅ Errores son rastreables (request ID → log → error response)
- ✅ Migraciones son repetibles (down/up limpio)
- ✅ Tests críticos pasan automáticamente

## 2. Multi-tenant Security 🔴

### 2.1 Problema

El tenant middleware (`src/middlewares/tenant.js`) inyecta `req.tenant` desde `req.user.store_id`, pero no todos los repositorios filtran por `store_id`. Un usuario autenticado de la tienda A podría —por error de código o manipulación de parámetros— acceder a datos de la tienda B.

### 2.2 Requisitos

1. **Tenant middleware auditado.** Verificar que:

   - `tenantContext` siempre se ejecuta **después** de `authenticate` (cuando la ruta requiere auth).
   - Cuando no hay `req.user` (rutas públicas como `GET /config`), se asigna `storeId: 1` (default).
   - No existe camino donde `req.tenant` quede `undefined`.

2. **Filtro universal por store_id.** Verificar repositorio por repositorio:

   | Repositorio | ¿Filtra por store_id? | ¿Por dónde recibe el tenant? |
   |-------------|----------------------|------------------------------|
   | `product.repository.js` | ✅ `baseQuery()` | `tenant.storeId` |
   | `category.repository.js` | ✅ `findAll()` | parámetro `tenantId` |
   | `user.repository.js` | ✅ `findAll()` | `tenant.storeId` |
   | `settings.repository.js` | ✅ `getPublicSettings()` | `storeId` |
   | `audit.repository.js` | ❌ `findAll()` | **no recibe tenant** — no filtra |

   El `audit.repository.findAll()` acepta `tenantId` pero **no lo usa en la query**. Se pasa como parámetro pero nunca se aplica como filtro `WHERE`. Esto es una vulnerabilidad multi-tenant.

   **⚠️ P0 — Bloqueante.** Este punto debe corregirse antes de cualquier otro. Si auditoría cruza tenants, rompe una de las garantías principales del sistema. Ningún método del repository puede ejecutar queries sin tenant context explícito.

3. **Cross-tenant injection test.** Crear pruebas que verifiquen que un usuario Store A **no puede**:

   - Ver productos de Store B (`GET /products/3` con ID de otro store → 404)
   - Editar categorías de Store B (`PUT /categories/1` → 404)
   - Listar usuarios de Store B (`GET /users` → solo usuarios de su store)
   - Ver auditoría de Store B (`GET /audit` → solo eventos de su store)

4. **Rutas públicas sin fuga.** Verificar que endpoints públicos no expongan datos multi-tenant:

   - `GET /api/v1/config` — solo settings públicas del store default (1)
   - `GET /api/v1/products/:slug/public` — solo productos del store 1 (hasta que haya ruteo por dominio)

### 2.3 Tests

| Test | Descripción | Expected |
|------|-------------|----------|
| `cross-tenant: products` | Store A intenta GET /products/:id de Store B | 404 |
| `cross-tenant: categories` | Store A PUT /categories/:id de Store B | 404 |
| `cross-tenant: users` | Store A lista usuarios, no incluye Store B | 0 resultados de B |
| `cross-tenant: audit` | Store A lista auditoría, solo eventos de Store A | 0 eventos de Store B |
| `cross-tenant: audit no tenant` | Query sin store_id es rechazada | Error o empty set |
| `unauthenticated: config` | GET /config sin token funciona | 200 con data |
| `unauthenticated: tenant` | req.tenant existe en ruta pública | storeId: 1 |

## 3. Auth Security

### 3.1 Refresh Token Rotation

El sistema actual implementa rotation pero debe validarse el flujo completo:

```
Login → accessToken + refreshToken (RT1)
  ↓
Refresh con RT1 → nuevo RT2 + nuevo AT (RT1 queda revoked)
  ↓
Refresh con RT1 (reuse detection) → ambos revocados → obligar re-login
  ↓
Logout con RT2 → RT2 revoked
  ↓
Refresh con RT2 → 401
```

### 3.2 Tabla refresh_tokens actual

La tabla existe como `sessions` en la migración. Verificar columnas:

| Columna | Tipo | ¿Existe? |
|---------|------|----------|
| `id` | SERIAL PK | ✅ |
| `user_id` | FK→users | ✅ |
| `token_hash` | VARCHAR(255) | ✅ |
| `expires_at` | TIMESTAMPTZ | ✅ |
| `revoked_at` | TIMESTAMPTZ NULL | ✅ |
| `created_ip` | VARCHAR(45) | ✅ |
| `user_agent` | TEXT | ✅ |

### 3.3 Password Policy

- Mínimo 6 caracteres (validado por Zod en auth.validator.js)
- Hash con Argon2id (memoryCost: 65536, timeCost: 3, parallelism: 2)
- Sin límite de caracteres superiores

### 3.4 Rate Limiting

- Login: 5 intentos por minuto por IP (ya verificado)
- Refresh: 10 por minuto (sugerido agregar)
- Logout: 10 por minuto (sugerido agregar)

## 6. Event System Hardening

### 6.1 Estado actual

El Event Bus (`src/events/index.js`) persiste eventos en `event_logs` y ejecuta handlers secuencialmente. Problemas identificados:

1. **Sin versionado.** El payload no incluye `version`, imposible migrar consumidores cuando cambia el schema.
2. **Eventos huérfanos.** Si el servidor se cae entre la inserción y la actualización del status, queda un evento `pending` que nunca se procesa.
3. **Sin replay.** No hay mecanismo para reprocesar eventos fallidos.
4. **Sin idempotencia.** Múltiples emisiones del mismo evento (por retry) pueden generar duplicados en audit_logs.

### 6.2 Schema de evento estándar

Todo evento debe incluir:

```json
{
  "name": "product.updated",
  "version": 1,
  "occurred_at": "2026-07-09T14:30:00.000Z",
  "producer": "products-service",
  "payload": { }
}
```

### 6.3 Contratos de eventos

Crear schemas de eventos en `src/events/contracts/`:

- `product.created.v1.json`
- `product.updated.v1.json`
- `product.deleted.v1.json`
- `user.login.v1.json`
- `user.logout.v1.json`

Cada contrato define el schema del `payload` usando Zod (reutilizable en validación y handlers).

### 6.4 Mejoras al Event Bus

1. **Agregar `version` al insert.** Modificar `emit()` para aceptar `version` y almacenarlo en `event_logs`. Requiere migración: `ALTER TABLE event_logs ADD COLUMN version INTEGER DEFAULT 1;`

2. **Replay endpoint.** Agregar `POST /api/v1/admin/events/:id/replay` que re-emite un evento existente (solo para administradores con permiso `events.replay`).

3. **Idempotencia en audit handler.** El handler de auditoría debe verificar si ya existe un audit_log con el mismo `event_log_id` antes de insertar.

4. **Worker de eventos fallidos.** El worker de jobs debe tener un contraparte para eventos: `startEventWorker()` que procesa eventos con status `failed` cada 60 segundos (con límite de reintentos).

### 6.5 Replay safeguards

El replay endpoint (`POST /api/v1/admin/events/:id/replay`) debe protegerse contra:

1. **Eventos antiguos.** No permitir replay de eventos con más de 7 días de antigüedad (configurable via `MAX_REPLAY_AGE_DAYS`).
2. **Versiones obsoletas.** Si el contrato del evento cambió (version mismatch), el replay debe rechazarse.
3. **Acciones externas.** Eventos que dispararon webhooks o integraciones externas no deben re-ejecutarse automáticamente. El replay ejecuta solo handlers internos (audit, logging), no webhooks.
4. **Idempotencia.** El replay debe generar un nuevo `event_log` con `id` fresco para evitar colisiones con el registro original.

Flujo:

```
Replay event #42
  ↓
validar que occurred_at < 7 días atrás
  ↓
validar que version del evento = versión actual del contrato
  ↓
crear nuevo event_log con payload original + meta.replayed_from = 42
  ↓
ejecutar handlers internos (NO webhooks)
  ↓
retornar nuevo event_log id
```

## 7. Job Queue Hardening

### 7.1 Estado actual

El job queue (`src/jobs/queue.js`) implementa enqueue, dequeue, complete, fail con exponential backoff. Problemas:

1. **Sin locking.** Dos workers pueden tomar el mismo job (carrera crítica entre `SELECT` y `UPDATE`).
2. **Sin dead letter queue.** Jobs que fallan permanentemente no tienen destino; se quedan como `failed` sin diagnóstico.
3. **Sin métricas.** No hay visibilidad de jobs procesados por hora, tasa de fallo, tiempo de procesamiento.

### 7.2 Mejoras

1. **Atomic dequeue con lock ownership.** Usar `UPDATE ... WHERE status = 'pending' ... RETURNING *` con subquery atomizada y columnas de ownership:

```sql
UPDATE jobs SET status = 'processing', started_at = NOW(),
  locked_by = 'worker-' || gen_random_uuid(),
  locked_at = NOW()
WHERE id = (
  SELECT id FROM jobs
  WHERE status = 'pending'
    AND available_at <= NOW()
  ORDER BY created_at ASC
  LIMIT 1
  FOR UPDATE SKIP LOCKED
)
RETURNING *;
```

Agregar columnas a la tabla `jobs`:

| Columna | Tipo | Descripción |
|---------|------|-------------|
| `locked_by` | VARCHAR(100) NULL | Identificador del worker que tomó el job |
| `locked_at` | TIMESTAMPTZ NULL | Cuándo se lockeó |

Esto evita que dos workers procesen el mismo job incluso si el `FOR UPDATE SKIP LOCKED` fallara.

2. **Dead letter queue.** Jobs que alcanzan `max_attempts` se mueven a `failed_jobs`:

| Columna | Tipo | Descripción |
|---------|------|-------------|
| `id` | SERIAL PK | |
| `original_job_id` | INTEGER | FK → jobs.id |
| `type` | VARCHAR(100) | |
| `payload` | JSONB | |
| `failed_reason` | TEXT | Último error |
| `attempts` | INTEGER | |
| `failed_at` | TIMESTAMPTZ | |

Crear migración 013 y modificar `fail()` para insertar en `failed_jobs` al alcanzar `max_attempts`.

3. **Admin endpoint para retry.** `POST /api/v1/admin/jobs/:id/retry` que re-encola un job fallido.

### 7.3 Tests

| Test | Descripción | Expected |
|------|-------------|----------|
| `jobs: enqueue` | Crear job → status: pending | ✅ |
| `jobs: dequeue` | Tomar job → status: processing | ✅ |
| `jobs: complete` | Marcar completado → status: completed | ✅ |
| `jobs: fail and retry` | Fallar job → status: pending con backoff | ✅ |

| `jobs: no double dequeue` | Dos dequeues simultáneos → solo uno obtiene el job | ✅ (atomic) |

## 8. Observabilidad

### 8.1 Correlation ID (end-to-end)

Renombrar conceptualmente `request ID` a `correlation_id`. Este ID debe viajar a través de todo el ciclo de vida de una operación:

```
HTTP Request → Header X-Correlation-Id o UUID generado
       ↓
  Logs (Pino reqId)
       ↓
  Events (event_logs.correlation_id)
       ↓
  Jobs (jobs.correlation_id)
       ↓
  Webhooks (webhook_events payload)
```

**Reglas:**
- Si el cliente envía `X-Correlation-Id`, se usa ese (permite trazabilidad desde el frontend).
- Si no, el middleware `requestId` genera uno nuevo (UUID v4).
- El `correlation_id` se propaga manualmente a eventos y jobs:

```javascript
// En un controller:
bus.emit('order.created', payload, {
  correlationId: req.correlationId,  // req.id
  // ...
});

// Enqueue un job con correlation_id:
enqueue('send-whatsapp', payload, new Date(), req.correlationId);
```

**Verificar:**
- Cada request tiene `req.id` único (UUID v4)
- Incluido en logs de Pino como `reqId`
- Retornado en `X-Request-Id` header
- Incluido en respuestas de error para trazabilidad
- `event_logs` persiste `correlation_id`
- `jobs` persiste `correlation_id`
- Un pedido puede rastrearse completo: API → Event → Job → Webhook

### 8.2 Logging

- Pino configurado con `pino-pretty` en desarrollo
- En producción: formato JSON para ingestión por sistema de logs
- Niveles: `info` (requests), `warn` (errores de negocio), `error` (excepciones internas)

### 8.3 Endpoint correlation_id en queue

Modificar `enqueue()` para aceptar `correlationId` como cuarto parámetro opcional:

```javascript
export async function enqueue(type, payload, availableAt = new Date(), correlationId = null) {
  // ...
  if (correlationId) jobData.correlation_id = correlationId;
  // ...
}
```

### 8.4 Métricas mínimas

```json
{
  "uptime": 3600,

  "active_jobs": 3,
  "failed_jobs_24h": 2,
```

### 9.1 Estructura de tests

Los tests se organizan por área en lugar de por tipo, con una carpeta dedicada para seguridad:

```
src/tests/
├── auth/             # Login, refresh, rotation, rate limit
├── tenant-isolation/ # Cross-tenant reads/writes/deletes (crítico)
├── api-contract/     # OpenAPI sync, response format, error codes
```

## 12. Criterios de aceptación

### Hardening: Multi-tenant

- [ ] `audit.repository.js` filtra por `store_id`
- [ ] Test cross-tenant products pasa (404)
- [ ] Test cross-tenant audit pasa (0 resultados)
- [ ] Test cross-tenant users pasa (0 usuarios de otro store)

### Hardening: Auth

- [ ] Refresh rotation test completo pasa
- [ ] Reuse detection test pasa (401 + ambos revocados)
- [ ] Logout invalida refresh token
- [ ] Rate limit en login, refresh, logout

### Hardening: API Contract

- [ ] OpenAPI documenta todos los endpoints existentes
- [ ] Formato de respuesta consistente (success/data/error/meta)
- [ ] Códigos de error estandarizados

### Hardening: Database

- [ ] Migración 013 aplica limpia (up/down)
- [ ] Migración 014 aplica limpia (up/down)
- [ ] Índices creados en tablas críticas
- [ ] FK products.category_id con ON DELETE SET NULL

### Hardening: Events

- [ ] Eventos incluyen `version`
- [ ] Replay endpoint funciona
- [ ] Audit handler no duplica eventos
- [ ] Eventos fallidos son recuperables

### Hardening: Jobs

- [ ] Dequeue atómico (sin double-fetch)
- [ ] Dead letter queue funciona
- [ ] Retry endpoint funciona

### Hardening: Observabilidad

- [ ] X-Correlation-Id / X-Request-Id en toda respuesta
- [ ] Error responses incluyen correlation_id
- [ ] `event_logs` persiste correlation_id
- [ ] `jobs` persiste correlation_id
- [ ] Client-provided correlation_id se respeta
- [ ] Métricas endpoint funcional

### Hardening: Tests

- [ ] `npm test` pasa completo
- [ ] Tests cross-tenant existen y pasan
- [ ] Tests de auth flow existen y pasan
- [ ] Tests de eventos existen y pasan
- [ ] Tests de jobs existen y pasan

## 13. Riesgos y mitigaciones

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| Cross-tenant data leak | Baja | 🔴 Crítico | Auditoría manual + tests automatizados |
| Refresh token rotation bug | Baja | 🟡 Alto | Tests de flujo completo |
| Migración rompe producción | Baja | 🟡 Alto | Test down/up en staging |
| Tests muy lentos | Media | 🟢 Bajo | Tests separados por área, timeout de 10s |
| Dead letter nunca se revisa | Alta | 🟢 Bajo | Dashboard admin con contador de failed_jobs |
