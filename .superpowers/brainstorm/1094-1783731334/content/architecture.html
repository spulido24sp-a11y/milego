Arquitectura: ¿Cómo construimos el Conversion Engine?

Tu spec es clara. Ahora hay que decidir CÓMO se implementa.

La pregunta clave

Toda landing debe construirse con bloques (Hero, Problem, Benefits, Testimonials, Offer, FAQ, CTA...).

¿Dónde se arma la landing?

A

Backend Engine (Node.js)

LIAM define la estructura → backend la renderiza como HTML en el servidor → se sirve como página estática. Cada bloque tiene schema + render en el backend. Performance máxima, SEO nativo. Los bloques se guardan en PostgreSQL como JSON.

Backend define bloques → Renderiza HTML → Sirve página completa
B

Frontend SPA + Editor Visual

Los bloques se renderizan en el navegador con vanilla JS. El admin tiene un editor visual drag-and-drop. La landing es una SPA que carga la configuración desde la API. Ideal para edición en vivo, pero más pesado y complejo.

Admin define bloques → API los sirve → Navegador los renderiza
C

Híbrido: Backend Render + Admin Editor

Recomendado. Backend renderiza HTML inicial (SSR-like con template strings) para performance y SEO. Admin tiene editor de bloques (genera JSON). LIAM puede modificar bloques vía API. Lo mejor de ambos mundos.

Admin define bloques → Backend renderiza HTML → Sirve página + API para live edit

MVP Scope

El spec completo es enorme (11 secciones, AI, editor visual, LIAM hipótesis, etc.).

¿Cuál debe ser el MVP?

1

MVP Conversional

Sistema de bloques funcional. LIAM genera JSON con bloques → engine renderiza landing completa. Sin editor visual. Sin drag-and-drop. LIAM decide todo. Puedes modificar el JSON en admin.

2

MVP Completo

Sistema de bloques + editor visual en admin + LIAM generación automática. Más ambicioso, más tiempo de desarrollo.

Mi recomendación: Opción C (Híbrido) + MVP Conversional (Opción 1). El backend ya existe y renderiza HTML estático. Los bloques se guardan como JSON en PostgreSQL. LIAM es quien genera/configura los bloques. El editor visual puede venir después.
